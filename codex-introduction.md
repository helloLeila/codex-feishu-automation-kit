# Codex 介绍

Codex 是一个面向软件开发与知识工作的 AI 编程助手。它可以阅读项目文件、理解代码结构、执行终端命令、修改文件、运行测试，并把这些操作串成一个完整的协作流程。和普通聊天机器人不同，Codex 更像一个能进入仓库工作的结对工程师：你提出目标，它负责探索上下文、制定修改方案、落地变更并验证结果。

## Codex 能做什么

Codex 最适合处理需要上下文理解和动手执行的任务，例如：

- 阅读代码仓库，解释模块之间的关系。
- 新建或修改文档、脚本、配置和源代码。
- 修复 bug，并通过测试或命令验证修复是否有效。
- 搭建小型应用、前端页面、工具脚本或项目原型。
- 根据现有风格补充功能，尽量保持代码一致性。
- 整理需求、写计划、生成 README、使用说明或技术文档。
- 辅助代码审查，指出潜在风险、回归点和测试缺口。

## 典型工作方式

一次高效的 Codex 协作通常会经历几个阶段：

1. 你描述目标，比如“帮我新增一个登录页面”或“解释这个仓库怎么启动”。
2. Codex 先查看项目结构、已有文件和相关命令，避免凭空猜测。
3. 如果需求还不够明确，Codex 会提出少量关键问题。
4. 在方向确定后，Codex 会直接修改文件或运行必要命令。
5. 修改完成后，Codex 会检查结果，并说明改了什么、如何验证。

这个流程的核心是：先理解上下文，再动手执行，最后给出可验证的结果。

## 适合交给 Codex 的任务

你可以把 Codex 用在以下场景：

- **学习项目**：让它帮你梳理目录结构、入口文件、数据流和核心逻辑。
- **日常开发**：让它补功能、改样式、写测试、修配置、处理报错。
- **文档建设**：让它写项目介绍、开发手册、接口说明、变更记录。
- **调试排查**：让它复现问题、定位原因、提出并验证修复。
- **重构辅助**：让它在不改变行为的前提下清理重复代码或改善结构。
- **原型探索**：让它快速实现一个可运行的想法，再逐步打磨。

## 使用建议

为了让 Codex 更准确地工作，可以尽量给出这些信息：

- 你想达成的结果，而不只是某个局部操作。
- 相关文件、命令、报错信息或截图。
- 你希望保持的风格、限制或技术栈。
- 是否需要它直接修改文件，还是先给你方案。
- 验收标准，比如“页面能打开”“测试通过”“文档适合新人阅读”。

例如，与其说“改一下这里”，不如说：

> 帮我把这个项目的 README 改成新人友好的启动文档，包含安装、运行、测试和常见问题。

这样的目标更清楚，Codex 也更容易一次做到位。

## 注意事项

Codex 很擅长处理工程上下文，但它仍然需要被正确使用：

- 对重要改动，最好让它运行测试或给出验证步骤。
- 对生产环境、支付、安全、隐私等高风险内容，需要额外审查。
- 如果仓库里有未提交改动，Codex 会尽量不覆盖用户已有修改。
- 如果需求涉及最新资料、第三方政策或在线服务状态，需要确认信息来源。
- 如果任务很大，先拆成小目标通常比一次性全做更稳。

## 快速开始

你可以从这些请求开始：

```text
帮我看一下这个仓库的结构，并告诉我怎么运行。
```

```text
帮我新建一个 README，介绍项目用途、安装方式和常用命令。
```

```text
这个测试失败了，帮我定位原因并修复。
```

```text
帮我把这个功能补完整，并运行验证命令。
```

```text
请以代码审查的方式检查最近的修改。
```

## 一句话总结

Codex 是一个能理解仓库、修改文件、运行命令并解释结果的 AI 开发协作者。它最有价值的地方，不只是回答问题，而是把“理解、实现、验证、说明”连成一条完整的工作链路。
